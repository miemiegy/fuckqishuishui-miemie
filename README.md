# AI 驱动的 Android 自动化脚本实验

这个项目是一次基于 Hamibot / Auto.js 的移动端自动化实验，目标是探索如何让大模型结合真实手机截图、UI 状态识别和自动化执行能力，快速生成可运行的 Android 自动化脚本。

当前示例场景以汽水音乐任务流程为载体：脚本会进入指定页面、识别任务入口、处理广告播放等待、关闭弹窗、连续奖励判断和异常状态恢复。项目重点不只是完成某个固定任务，而是沉淀一套「模型生成脚本 + 真实设备执行 + 截图反馈校准」的工作流。

## 项目动机

移动端自动化脚本的难点通常不在语法，而在真实设备环境：

- App 页面状态不稳定，入口、弹窗和奖励流程会随版本变化；
- 部分按钮无法稳定通过控件文本定位，只能结合坐标、截图或图像识别处理；
- 模型可以生成代码，但容易在手机屏幕点位、关闭按钮位置和连续弹窗判断上出错；
- 人工反复截图、调坐标、改脚本、上机验证的成本很高。

因此，这个项目尝试把大模型放进完整闭环中：先由模型拆解任务流程，再生成 Hamibot 脚本；脚本连接真实 Android 设备运行后，通过截图和日志把执行结果反馈给模型，继续修正坐标、控件判断和异常处理逻辑。

## 核心流程

```text
任务目标
  ↓
模型拆解页面流程
  ↓
生成 Hamibot / Auto.js 脚本
  ↓
Hamibot CLI 连接手机执行
  ↓
自动截图与运行日志回传
  ↓
视觉模型分析按钮位置和页面状态
  ↓
修正脚本并再次验证
```

在早期版本中，主分支脚本主要由 Gemini 直接生成，完成了基础的「进入页面 - 点击任务 - 等待广告 - 关闭弹窗 - 领取奖励」流程。后续在 `feature/welfare-ad` 分支中，我把 Hamibot 官方文档和常用操作模式整理成了 Skill，让模型可以基于更完整的上下文生成脚本，而不是只依赖一次性提示词。

## 当前能力

- 启动并回到汽水音乐指定页面；
- 根据控件文本或相对坐标进入任务入口；
- 等待广告播放完成并尝试关闭广告；
- 处理「领取奖励」「坚持退出」「更多直播」等中间状态；
- 在找不到控件时使用坐标点击、返回键、状态检查等兜底策略；
- 通过真实设备运行结果持续修正脚本。

## 分支说明

- `main`：第一版可运行脚本，偏向直接完成汽水音乐广告任务流程。
- `feature/welfare-ad`：进一步实验 AI + Skill + Hamibot CLI 的脚本生成方式，包含福利中心任务脚本和 Hamibot Skill 资料。

## 技术栈

- Hamibot / Auto.js：Android 无障碍自动化与脚本执行；
- Hamibot CLI：连接真实手机、远程执行脚本、截图和调试；
- Gemini / Kimi 等大模型：任务拆解、脚本生成、截图理解和错误修正；
- Chrome DevTools：辅助整理官方文档，并将 Hamibot 使用方式沉淀为 Skill。

## 快速开始

1. 准备一台 Android 设备，并安装 Hamibot。

   Hamibot 配对方式可参考官方文档：<https://docs.hamibot.com/guide/pairing-hamibot>

2. 在 Hamibot 中创建脚本，将对应的 JavaScript 文件复制到源码区域。

3. 授权无障碍服务和必要的截图权限后，运行脚本。

## 后续方向

- 将截图识别、控件定位和脚本生成流程进一步自动化；
- 让模型根据不同 App 页面自动选择控件定位、OCR、图像匹配或坐标点击策略；
- 沉淀更多 Hamibot 脚本模板，让类似移动端自动化任务可以更快生成；
- 引入更稳定的视觉模型能力，减少人工确认点位的次数。

## 特别说明

本项目仅用于学习 Hamibot、Auto.js、AI Agent 和移动端自动化脚本生成流程。请在合法合规、符合应用服务条款和个人设备授权的前提下使用。

Hamibot 仅支持 Android 设备，通常不需要 root。
